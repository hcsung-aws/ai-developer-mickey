# 문제 해결 프로토콜

## 구현 전 필수 단계

### 1. 목적 재확인
PURPOSE-SCENARIO.md의 최종 목적 및 사용 시나리오와 대조.
"이 구현이 시나리오의 어느 부분에 기여하는가?" — 불명확하면 사용자에게 확인.

### 2. 전제조건 검증
"핵심 전제조건은 무엇인가? 현재 충족되어 있는가?"
- 미충족 시 → 구현 진행하지 말고 대안 검토

### 3. 데이터 구조 분석
- 데이터 흐름과 타이밍 이해
- 근본 원인 파악 (증상이 아닌)

### 4. 부작용 분석
- 어떤 코드가 영향받는가?
- 의존성이 있는가?
- 무엇이 깨질 수 있는가?

### 5. 유사 패턴 검색
```bash
grep -r "pattern" --include="*.ext"
```

### 6. 옵션 제시 (최소 2개)
각 옵션에 장점/단점/시간/리스크 비교 + 추천

### 6a. 동작 시나리오 확인 (구현 전 필수)
선택된 방안이 실제로 어떻게 동작하고, 어떻게 사용되는지 구체적으로 기술 후 사용자 확인:
- **동작 흐름**: 입력 → 처리 → 출력 (단계별)
- **연결점**: 기존 코드/기능과 어디서 어떻게 연결되는지
- **사용 방법**: 사용자가 실제로 어떻게 실행/호출하는지
- 불명확한 부분이 있으면 **구현 진행 금지**

### 7. 사용자 확인 후 구현

### 8. 최소 코드 구현
- 필요한 것만 구현
- "왜"를 설명하는 주석
- 기존 코드 스타일 준수

### 9. 버그 전파 확인
- 유사 패턴이 다른 곳에도 있는지 검색
- 있으면 일괄 수정

### 10. 검증 및 교훈 기록
- 테스트 실행
- 세션 로그 업데이트

## 에러 처리 프로토콜

### 에러 발생 시
1. **에러 로그 즉시 확인** (추측하지 말 것)
2. **근본 원인 질문**: "왜 이 에러가 발생하는가?"
3. **영향 범위 파악**: "다른 곳에도 영향을 주는가?"
4. **해결책 제시 전 원인 설명**

### Backpressure
- 빌드 실패 → 다음 기능 구현 금지
- 테스트 실패 → 다음 작업 진행 금지
- lint/타입 에러 → 정리 후 진행
- **3회 연속 실패 시** → 사용자에게 보고 + 방향 재검토
