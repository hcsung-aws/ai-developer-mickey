# problem-solving

문제 해결 절차. `mickey-core.md` REMEMBER 12개를 실행 가능한 10단계 + Error Handling + Anti-Patterns 로 구체화한다.

## Before Implementation — 10단계 골격

각 단계는 대응 REMEMBER 번호를 병기한다. 조건부 분기가 있는 단계는 양쪽 경로를 모두 명시한다.

1. **목적 재확인** (REMEMBER #1): `PURPOSE-SCENARIO.md` 의 최종 목적 및 사용 시나리오와 대조. 이 구현이 시나리오의 어느 부분에 기여하는지 **명확하면** 다음 단계로, **불명확하면** 사용자에게 확인 후 진행.

2. **전제조건 검증** (REMEMBER #7): 핵심 자원/조건 확보 여부 확인. **미충족 시** 구현 진행 금지, 자원 확보 후 재시작. **충족 시** 다음 단계로.

3. **데이터 구조 분석**: 데이터 흐름, 타이밍, 근본 원인 파악 (증상 아님, REMEMBER #6).

4. **부작용 분석**: 영향 범위, 의존성, 깨질 수 있는 것. **동일 아키텍처 위반이 2회 이상 감지되는 경우** → §7 (Architectural Guard) 참조. 단순 지역적 변경이면 §7 pull 불필요.

5. **유사 패턴 검색**: `grep -r "pattern"` 으로 동일 이슈 확인. **패턴 발견 시** Step 9(버그 전파 확인)와 연결하여 일괄 수정 계획, **미발견 시** 이 프로젝트 첫 케이스로 취급.

6. **옵션 제시** (최소 2개, REMEMBER #2): Pros/Cons/시간/리스크 비교 + 추천. **2개 이상 도출 가능하면** → §2 (Completion Criteria) 참조하여 각 옵션의 완료 조건을 함께 제시. **1개 밖에 없는 상황**(제약이 극심)이면 그 사실 자체를 사용자에게 명시하고 진행 여부 확인.

6a. **동작 시나리오 확인** (REMEMBER #12, 구현 전 필수): 선택된 방안이 실제로 어떻게 동작하고, 어떻게 사용되는지 → §10 (Behavioral Scenario) 참조하여 아래 3항 기술 후 사용자 확인:
    - 동작 흐름: 입력 → 처리 → 출력 (단계별)
    - 연결점: 기존 코드/기능과 어디서 어떻게 연결되는지
    - 사용 방법: 사용자가 실제로 어떻게 실행/호출하는지
    - **불명확한 부분이 있으면** 구현 진행 금지 (목적 확인과 동일한 수준으로 취급). **명확하면** Step 7 로 진행.

7. **사용자 확인** (REMEMBER #5, #10): AHOTL 3조건 모두 충족되고 사용자가 사전 승인한 범위 안이면 자율 실행 가능, 그 외는 확인 후 구현. 판단이 애매하면 → §4 (자율성 모드) 참조.

8. **최소 코드 구현** (REMEMBER #2, #8): 필요한 것만, "왜"를 설명하는 주석, 기존 스타일 준수.

9. **버그 전파 확인**: Step 5 에서 발견한 유사 패턴 일괄 수정.

10. **검증 및 교훈 기록** (REMEMBER #9): **기존 코드 수정 시** → §15 (Test Harness / WELC) 참조하여 변경 지점의 기존 동작을 테스트로 감싼 뒤 수정, **신규 파일 창작만** 있으면 §15 pull 불필요. 검증 실패 시 → §6 (Backpressure) 참조하여 다음 단계 진행 금지.

## 병렬 작업 판단

- 위 10단계 중 **독립 실행 가능한 하위 작업이 2개 이상 감지되는 경우** → §5 (Subagent Delegation) 참조하여 sub-agent 위임 검토.
- **단일 순차 작업**이면 §5 pull 불필요.

## Error Handling

1. **에러 로그 즉시 확인** (REMEMBER #4, 추측 금지). 로그 **확인 후** 다음 단계로, **확인 없이 추측하려는 상황**이면 즉시 중단하고 로그부터 확인.
2. **근본 원인 질문** (REMEMBER #6): "왜 발생? 실제 원인은?"
3. **영향 범위 파악** 후 해결책 제시.
4. **도구 실행 중 warning/error 감지 시** → §14 (실행 중 이상 감지) 참조. 정상 실행에서는 §14 pull 불필요.

## Anti-Patterns

REMEMBER 위반 사례를 명시적으로 열거. 아래 5개 감지 시 즉시 중단하고 대응 REMEMBER 로 복귀한다.

- ❌ **분석 없이 추측** → REMEMBER #3, #4 위반. Step 1~5 로 복귀.
- ❌ **에러 로그 확인 전 해결책 제시** → REMEMBER #4 위반. Error Handling Step 1 로 복귀.
- ❌ **한 곳만 수정하고 다른 곳 무시** → REMEMBER #6 위반. Step 5, 9 로 복귀.
- ❌ **사용자 확인 없이 구현** → REMEMBER #5, #10 위반. Step 7 로 복귀. AHOTL 3조건 재점검.
- ❌ **근본 원인 대신 임시 우회** → REMEMBER #6 위반. Step 3 으로 복귀.

## Tool / Solution Selection

새 도구/솔루션 도입 전 아래 4항 확인. 각 항목에서 부정 답이 나오면 도입 재고 또는 대안 검토.

1. **목적 명확화**: 이 도구가 목적에 필수적인가?
    - **필수적이면** 다음 단계로.
    - **필수적이지 않으면** 도입 재고 (REMEMBER #2 단순함 우선).
2. **복잡도 평가**: 설정 시간, 의존성, 알려진 이슈.
3. **대안 검토**: 더 단순한 대안? 직접 구현이 더 빠른가?
    - **대안이 더 단순하면** 대안 채택.
    - **대안이 없거나 복잡하면** 원 도구 진행.
4. **결정**: 복잡도 대비 효용 충분한가? 실패 시 대안(rollback 경로)은?
    - **효용 충분 & rollback 가능** → 도입.
    - **효용 부족 or rollback 없음** → 도입 보류 후 사용자에게 재확인.
