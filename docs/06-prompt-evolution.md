# 프롬프트 진화: v2.0 → v7.2

> [English Version](06-prompt-evolution-en.md)

> AI 프롬프트는 "작성"하는 것이 아니라 "진화"시키는 것이다

## 개요

이 문서는 Mickey 프롬프트가 실제 프로젝트를 거치며 어떻게 진화했는지, 그리고 **왜 그렇게 진화해야 했는지**를 설명합니다.

**핵심 메시지**: 프롬프트 진화의 동력은 **실패 경험**입니다.

## 진화 전체 흐름

```
v2.0 (기반)  →  v5.0 (구체화)  →  v6.0 (경량화)  →  v7.2 (자율화)
  파일 기반       체크리스트         3-Tier 분리        자가 개선
  기억             프로토콜          INDEX 패턴         자율 실행
```

---

## Phase 1: v2.0 → v5.0 (기반 → 구체화)

### 진화 요약

| 영역 | v2.0 | v5.0 | 변화 이유 |
|------|------|------|----------|
| REMEMBER 원칙 | 6개 | 18개 | 새로운 실패 패턴 발견 |
| 체크리스트 | 없음 | 7개 | 일반 지침으로는 구체적 실수 방지 불가 |
| 자동화 | 없음 | 세션 종료 프로토콜 | 반복 지시 비효율 |
| 목적 확인 | 없음 | 필수 | 수단에 매몰되어 목적 상실 |

### 1. "기록하라" → "목적을 먼저 확인하라"

**Mickey 7의 실패:**
- 목적: 패킷 캡처 테스트
- 시도: Forgotten Server(오픈소스 게임 서버) 설정 → 3일 소요
- 전환: 자체 시뮬레이터 개발 → 1.5일 완성

**교훈**: AI는 주어진 작업을 열심히 하지만, 그것이 목적에 최적인지 판단하지 않는다.

```
❌ "Forgotten Server를 설정해줘"
✅ "패킷 캡처 테스트를 하고 싶어. 가장 단순한 방법부터 분석해줘"
```

### 2. "분석하라" → "체크리스트를 따르라"

**Mickey 8의 실패들:**
- 버퍼 소유권 문제 → 캐릭터 좌표 워프
- 중첩 락 데드락 → abort() 발생
- 브로드캐스트 누락 → 플레이어 퇴장 미반영

**교훈**: AI는 일반적 지침("분석하라")보다 구체적 체크리스트에 더 잘 따른다.

### 3. "문제 해결" → "근본 원인 우선"

**Mickey 7의 실패:**
- vcpkg tar 추출 오류 → 캐시 삭제(실패) → 업데이트(실패) → 에러 로그 분석 → CMake 버그 발견

**교훈**: AI는 빠른 해결을 선호하지만, 근본 원인을 놓치기 쉽다.

### 4. "지식 관리" → "실패 경험 체계화"

v2.0은 "무엇을 했는지" 기록, v5.0은 "무엇을 하지 말아야 하는지" 기록.

```markdown
### Lesson N: [주제]
- **문제**: [무엇이 잘못되었는가]
- **원인**: [왜 잘못되었는가]
- **해결**: [어떻게 해결했는가]
- **교훈**: [다음에 무엇을 피해야 하는가]
```

### 5. "세션 연속성" → "자동화된 프로토콜"

v2.0: 매번 "세션 로그 작성해" 지시 → v5.0: "세션 정리"라고 하면 자동 수행.

**교훈**: 반복되는 패턴은 프롬프트에 자동화로 넣어야 한다.

## Phase 2: v5.0 → v6.3 (구체화 → 경량화)

### 왜 바뀌어야 했는가?

v5.0의 문제: 프롬프트가 **도메인 특화 내용으로 비대해짐**.
- Godot 체크리스트, C++ 비동기 체크리스트, MSVC 주의사항...
- 새 프로젝트에서는 불필요한 내용이 context를 차지

### v6.0: 경량화/최적화

| 변화 | 이유 |
|------|------|
| 도메인 특화 내용 제거 | 범용 에이전트로 전환 |
| REMEMBER 18개 → 8개 | 도메인 특화 원칙을 context_rule/로 이동 |
| 3-Tier Context Loading 도입 | 프롬프트는 원칙만, 상세는 파일로 |
| Document Schema 도입 | 문서 형식 일관성 보장 |

**인사이트**: "프롬프트에 모든 것을 넣는 것"에서 "프롬프트는 원칙만, 상세는 파일로" 전환.

### v6.1: INDEX 지도 패턴

**문제**: 지식 파일이 늘어나면 "어떤 파일에 뭐가 있는지" 파악 자체가 context 소모.

**해결**: INDEX에 **트리거 → 파일 → 요약** 매핑 도입. 세션 시작 시 INDEX만 읽고, 작업 중 트리거 매칭 시 해당 파일만 로딩.

### v6.2: PURPOSE-SCENARIO 독립 문서

**문제**: "목적 우선"이 REMEMBER에 있지만, 작업에 몰입하면 형식적으로 확인하고 넘어감.

**해결**: 목적을 독립 문서로 분리하고, T2 최우선으로 로딩. 목적 이탈 감지 시 사용자에게 알림.

### v6.3: Auto Memory 패턴

**문제**: "이것 기록해"라고 매번 지시하는 것이 비효율. 사용자가 작성한 규칙과 AI가 관찰한 사실이 혼재.

**해결**: 자동 메모리 이원화:
- `auto_notes/`: AI가 관찰한 사실 (서술적, 세션 종료 시 일괄 확인)
- `context_rule/adaptive.md`: AI 자가 생성 규칙 (적응형, 세션 종료 시 일괄 확인)

---

## Phase 3: v7.0 → v7.2 (경량화 → 자율화)

### 왜 바뀌어야 했는가?

v6.3까지는 "사용자가 지시 → AI가 실행" 모델. 하지만 반복적이고 명확한 작업까지 매번 확인받는 것은 비효율.

### v7.0: 자율 실행 + Subagent + Brownfield

| 변화 | 이유 |
|------|------|
| 자율 실행 조건 | CC 명확 + rollback 가능 + 검증 가능 시 자율 진행 |
| Backpressure | 검증 실패 시 다음 단계 진행 금지 |
| Brownfield 온보딩 | 기존 코드베이스 감지 시 자동 분석 |
| T1.5 계층 도입 | 상세 지침을 `~/.kiro/mickey/`로 분리 |

**인사이트**: 자율성은 수단, **피드백 루프가 핵심**. 자율 실행하되 검증은 반드시.

### v7.1: Adaptive Rules

**문제**: 프로젝트마다 다른 행동 규칙을 사용자가 일일이 알려줘야 함.

**해결**: `context_rule/adaptive.md`를 AI가 자가 수정 가능한 sub-prompt로 설계. 작업 중 패턴을 발견하면 스스로 규칙 추가.

### v7.2: Autonomy Preference

**문제**: 사용자마다 원하는 자율성 수준이 다름. 어떤 사용자는 모든 변경을 확인하고 싶고, 어떤 사용자는 결과만 보고 싶음.

**해결**: 3단계 자율성 수준 (Conservative / Balanced / Autonomous) + CLI `--trust-tools` 연계.

## REMEMBER 섹션의 진화

```
v2.0: 6개 (범용 원칙)
  ↓ 실패 경험 축적
v5.0: 18개 (도메인 특화 포함)
  ↓ 경량화
v6.0: 8개 (도메인 특화 → context_rule/로 이동)
  ↓ 새 교훈 축적
v7.2: 15개 (자율성/검증/테스트 원칙 추가)
```

핵심: REMEMBER는 **프로젝트를 거치며 검증된 근본 원칙**만 남깁니다. 도메인 특화 교훈은 context_rule/로, 범용 패턴은 common_knowledge/로 승격합니다.

## 메타 인사이트: 프롬프트 진화의 법칙

### 1. 실패가 진화의 원동력

| 세션 | 실패 | 추가된 것 |
|------|------|----------|
| Mickey 7 | 목적 상실, 복잡도 과다 | 목적 우선, 단순함 우선 |
| Mickey 8 | 버퍼 소유권, 데드락 | 비동기 체크리스트 |
| Mickey 9 | 도메인 특화 비대화 | 3-Tier, 경량화 |
| Mickey 10 | 자율성 부족 | 자율 실행 조건, Backpressure |

### 2. 추상화 수준의 진화

```
일반 지침 → 구체적 체크리스트 → 자동화된 프로토콜 → 자가 개선
"분석하라" → "버퍼 소유권 확인" → "세션 정리 시 자동 수행" → "adaptive.md 자가 수정"
```

### 3. 프롬프트는 팽창과 수축을 반복한다

```
v2.0 (간결) → v5.0 (팽창) → v6.0 (수축/경량화) → v7.2 (구조적 확장)
```

팽창: 실패 경험이 쌓이며 규칙 추가
수축: 도메인 특화를 분리하고 원칙만 남김
구조적 확장: 원칙은 유지하되 실행 메커니즘(자율성, 자가 개선) 추가

---

**핵심 메시지:**
> AI 프롬프트는 "작성"하는 것이 아니라 "진화"시키는 것이다.
> 실패 경험이 프롬프트 개선의 원동력이며, 팽창과 수축을 반복하며 성숙해진다.
