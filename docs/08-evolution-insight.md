# AI를 잘 쓰는 법의 진화: Mickey가 발견한 것들

> Mickey 에이전트를 만들고 개선하면서 발견한 "AI를 효과적으로 활용하는 법"이 어떻게 진화해 왔는지, 그리고 어디로 향하고 있는지에 대한 기록.

## 이 문서를 읽어야 하는 이유

AI 코딩 어시스턴트를 써본 사람이라면 이런 경험이 있을 것입니다:

- 어제 설명한 것을 오늘 다시 설명해야 한다
- 작업에 몰입하다 보면 원래 목적에서 벗어난다
- 프로젝트가 커질수록 AI가 전체 그림을 놓친다
- 기존 코드베이스에서는 AI가 유독 약하다

Mickey는 이런 문제들을 하나씩 부딪히며 해결해 온 AI 개발 에이전트입니다. 이 문서는 그 과정에서 발견한 패턴들을 정리한 것으로, "AI를 잘 쓰는 법"이 단순한 프롬프트 기법이 아니라 **시스템 설계의 문제**라는 것을 보여줍니다.

---

## Phase 1: 기억의 문제 (v2.0, 2024-12)

### 원인: AI는 기억하지 못한다

Godot 리플레이 시스템을 개발하면서 발견한 첫 번째 문제는 단순했습니다. AI 어시스턴트는 세션이 끝나면 모든 것을 잊습니다. 어제 논의한 설계 결정, 발견한 버그, 합의한 방향 — 전부 사라집니다.

### 결과: 세션 연속성 시스템

파일 기반 메모리를 도입했습니다:
- **SESSION.md**: 각 세션의 작업 기록
- **HANDOFF.md**: 다음 세션에 전달할 핵심 정보
- **context_rule/**: 프로젝트별 규칙 (반복 실패 방지)
- **common_knowledge/**: 프로젝트 무관 재사용 패턴

### 발견한 원칙

> **"AI의 기억은 파일이다."** 대화는 휘발되지만, 파일은 남는다. 세션 간 연속성은 구조화된 파일로만 보장된다.

이것은 나중에 Harness Engineering에서 "repository knowledge as the system of record"라는 이름으로 재발견됩니다.

---

## Phase 2: 방향의 문제 (v5.0, 2025-01)

### 원인: AI는 방향을 잃는다

패킷 캡처 에이전트를 개발하면서 발견한 두 번째 문제는 더 미묘했습니다. AI는 주어진 작업을 잘 수행하지만, **왜 이 작업을 하는지**를 잊습니다. 구현에 몰입하면 원래 목적에서 벗어나고, 복잡한 솔루션을 선택하고, 근본 원인 대신 증상을 치료합니다.

### 결과: 목적 우선 원칙 + 체크리스트

5가지 핵심 원칙을 도입했습니다:
1. 목적 우선 — 모든 판단의 기준
2. 단순함 우선 — 복잡한 솔루션보다 단순한 대안
3. 근본 원인 우선 — 증상이 아닌 원인 해결
4. 분석 후 구현 — 추측 금지
5. 사용자 확인 후 변경 — 독단 금지

v5.1~v5.4에서는 실전 프로젝트(AI Agent 자동화 플랫폼)를 거치며 교훈이 계속 추가되었습니다:
- v5.1: 전제조건 검증 (구현 전 핵심 자원 확보 확인)
- v5.2: 점진적 도입 (최소 기능 시작 + 피드백 기반 확장)
- v5.3: 세션 종료 프로토콜 (교훈 자동 분류 + 개선 제안)
- v5.4: 필수 테스트 프로토콜 (테스트 없이 완료 선언 금지)

### 발견한 원칙

> **"원칙은 실패에서 태어난다."** 각 REMEMBER 항목 뒤에는 실제 실패 경험이 있다. v5.0의 24개 원칙은 24번의 실패에서 추출된 것이다.

---

## Phase 3: 효율의 문제 (v6.0-6.2, 2026-02)

### 원인: 정보가 많다고 좋은 게 아니다

v5.x까지 누적된 프롬프트는 도메인 특화 내용(C++ 패턴, 게임 서버 체크리스트, MSVC 빌드 규칙)으로 비대해졌습니다. 새 프로젝트에서는 쓸모없는 정보가 context window를 차지하고, AI가 관련 없는 규칙에 주의를 빼앗겼습니다.

### 결과: 3단계 경량화

**v6.0 — 구조 혁신**: 도메인 특화 내용을 전부 제거하고, 3-Tier Context Loading을 도입했습니다.
- T1(항상): 범용 원칙만 (시스템 프롬프트)
- T2(세션 시작): 프로젝트 핵심 정보만
- T3(필요 시): 특정 지식 파일만

**v6.1 — INDEX 지도 패턴**: T3를 더 세분화했습니다. INDEX.md를 "지식의 목차"로 사용하여, 세션 시작 시 "어떤 지식이 있는지"만 파악하고, 실제 작업 중 필요한 파일만 로딩합니다.

**v6.2 — PURPOSE-SCENARIO**: "목적"을 체크리스트 항목이 아닌 독립 문서로 분리했습니다. 모든 판단의 최우선 기준으로 격상하고, 작업 중 목적과의 정합성을 지속적으로 체크합니다.

### 발견한 원칙

> **"지도를 주되, 백과사전을 주지 마라."** AI에게 필요한 것은 모든 정보가 아니라, 어디에 어떤 정보가 있는지 아는 것이다. Progressive disclosure — 작은 진입점에서 시작하여 필요할 때 깊이 탐색.

이것은 Harness Engineering에서 "AGENTS.md를 목차로 쓰고, 상세는 docs/에 분산"하는 패턴과 정확히 일치합니다. Mickey는 이 패턴을 독립적으로 발견한 셈입니다.

## Phase 4: 기록의 문제 (v6.3, 2026-03)

### 원인: 사용자가 모든 것을 지시해야 한다

Claude Code의 Auto Memory 기능을 분석하면서 발견한 문제입니다. Mickey는 교훈을 잘 정리하지만, **사용자가 "이것을 기록해"라고 말해야만** 기록합니다. 빌드 명령, 파일 역할, 에러 해결법 같은 관찰 사실은 AI가 스스로 기록할 수 있는데, 매번 사용자 확인을 받는 것은 비효율적입니다.

### 결과: 자동 메모리 이원화

"사용자가 작성하는 규칙"과 "AI가 기록하는 관찰 사실"을 분리했습니다:
- **auto_notes/**: AI가 자동 기록하는 관찰 사실 (세션 종료 시 일괄 확인)
- **context_rule/**: 사용자가 검증한 규칙 (즉시 확인)
- **common_knowledge/**: 범용 패턴 (즉시 확인)

파일 크기 제한, 작업 단위 트리거, 교훈 승격 경로도 함께 도입했습니다.

### 발견한 원칙

> **"저위험 관찰은 자동으로, 고위험 결정은 사용자와 함께."** 모든 것에 사용자 확인을 받는 것은 안전하지만 느리다. 위험도에 따라 자율성 수준을 구분하면 속도와 안전을 모두 확보할 수 있다.

## Phase 5: 자율성과 협업의 문제 (v7 계획, 2026-03)

### 원인: 단일 에이전트의 한계

2026년 AI 개발 트렌드(Harness Engineering, AI-DLC, Ouroboros)를 조사하면서 발견한 문제입니다. Mickey는 단일 에이전트로 분석/구현/검증을 모두 수행합니다. 이것은 세 가지 한계를 만듭니다:

1. **속도**: 독립적인 작업도 순차 처리해야 한다
2. **깊이**: 한 모드에 몰입하면 다른 관점을 놓친다
3. **규모**: 복잡한 프로젝트에서 단일 에이전트의 context window가 부족하다

### 트렌드에서 배운 것

**Harness Engineering** (OpenAI): 인간은 코드를 쓰지 않고, 에이전트가 실행할 수 있는 환경(harness)을 설계합니다. repo가 지식의 system of record이고, 결정론적 린터와 구조 테스트로 아키텍처를 강제하며, 가비지 컬렉션 에이전트로 엔트로피를 관리합니다.

**AI-DLC**: Hat 기반 역할 분리(Planner/Builder/Reviewer)와 Completion Criteria로 자율성을 확보합니다. 핵심 공식은 `자율성 = f(기준 명확도)` — 완료 기준이 명확할수록 AI가 자율적으로 실행할 수 있습니다. Backpressure(제약 기반 품질 게이트)로 "이렇게 해라" 대신 "이 조건을 만족시켜라"고 지시합니다.

**Ouroboros**: 자기 수정 AI 에이전트로, supervisor/worker 모델로 작업을 분해하고 병렬 실행합니다. Background consciousness로 작업 사이에 능동적으로 사고하며, multi-model review로 자기 변경을 검증합니다.

### 결과: 자율 실행 + 피드백 루프

Mickey v7의 방향성은 "자율성은 수단, 피드백 루프가 핵심"입니다:

- **Completion Criteria**: 각 작업에 검증 가능한 완료 기준을 명시. 기준이 명확하면 자율 실행, 불명확하면 사용자와 협의
- **Subagent Delegation**: Mickey가 orchestrator로서 목적별 subagent에 작업 위임 (탐색/구현/검증/지식관리)
- **자율성 모드**: HITL(고위험) / OHOTL(창의적) / AHOTL(기계적) 구분
- **엔트로피 자동 관리**: 세션 시작 시 지식 베이스 자동 정리

### 발견한 원칙

> **"자율 실행의 가치는 속도가 아니라 교훈이다."** Ouroboros는 자율 실행 자체가 목적이지만, Mickey는 자율 실행의 결과에서 교훈을 추출하여 다음 세션에 반영하는 것이 목적이다. 빠르게 실행하고, 결과에서 배우고, 배운 것을 구조화하여 다음에 더 잘한다.

## 핵심 발견: Brownfield에서의 점진적 Harness 구축

모든 진화 과정을 관통하는 하나의 발견이 있습니다.

Harness Engineering, AI-DLC, Ouroboros는 모두 **Greenfield(신규 개발)에 최적화**되어 있습니다. 처음부터 깨끗한 환경에서 harness를 설계하거나, 자기 코드를 처음부터 생성하거나, Intent를 깔끔하게 분해합니다.

하지만 현실의 대부분의 프로젝트는 **Brownfield(기존 코드 개선)**입니다. 기존 코드가 있고, 암묵적 규칙이 있고, 기술 부채가 있습니다. Martin Fowler 사이트의 Birgitta Böckeler는 이렇게 지적합니다: *"기존 코드베이스에 harness를 retrofit하는 것은, 한 번도 static analysis를 돌리지 않은 코드베이스에 처음 돌리는 것과 같다."*

Mickey의 진화 과정이 자연스럽게 이 문제의 해답을 제시합니다:

```
Harness Engineering:  처음부터 완전한 harness를 설계 (Greenfield 최적)
Mickey:               세션이 쌓일수록 harness가 두꺼워짐 (Brownfield 적합)

1세션: 환경 스캔 + 기본 구조 파악
2세션: 핵심 파일 역할 + 빌드 명령 축적
3세션: 의존성 패턴 + 반복 실패 규칙 축적
...
N세션: 프로젝트 전문가 수준의 지식 베이스 완성
```

이것이 Mickey의 고유한 포지션입니다. 세션 연속성(Phase 1) + 목적 우선(Phase 2) + 효율적 구조화(Phase 3) + 자동 기록(Phase 4) + 자율 실행(Phase 5)이 결합되면, **Brownfield에서도 세션이 쌓일수록 점점 더 정교해지는 AI 개발 에이전트**가 됩니다.

## 진화의 전체 그림

| Phase | 버전 | 발견한 문제 | 해결 방법 | 핵심 원칙 |
|-------|------|-----------|----------|----------|
| 1 | v2.0 | AI는 기억하지 못한다 | 파일 기반 세션 연속성 | AI의 기억은 파일이다 |
| 2 | v5.0-5.4 | AI는 방향을 잃는다 | 목적 우선 + 실패 기반 원칙 | 원칙은 실패에서 태어난다 |
| 3 | v6.0-6.2 | 정보 과잉은 해롭다 | 3-Tier 로딩 + INDEX 지도 | 지도를 주되, 백과사전을 주지 마라 |
| 4 | v6.3 | 모든 것에 확인을 받는 것은 느리다 | 자동 메모리 이원화 | 저위험은 자동, 고위험은 함께 |
| 5 | v7 계획 | 단일 에이전트의 한계 | 자율 실행 + subagent 협업 | 자율 실행의 가치는 교훈이다 |

각 Phase는 이전 Phase의 기반 위에 쌓입니다. 기억이 없으면 방향을 잡을 수 없고, 방향이 없으면 구조화가 무의미하고, 구조가 없으면 자동 기록이 혼란을 만들고, 자동 기록이 없으면 자율 실행의 교훈이 사라집니다.

## 앞으로의 방향

Mickey v7의 구체적 개선 계획은 [IMPROVEMENT-PLAN-v7.md](IMPROVEMENT-PLAN-v7.md)에 정리되어 있습니다.

핵심 방향은 세 가지입니다:

1. **Brownfield 온보딩 프로토콜**: 기존 코드베이스에서 빠르게 지식을 추출하는 구조화된 탐색 절차
2. **Completion Criteria + 자율 실행**: 명확한 완료 기준 내에서 자율 진행, 결과에서 교훈 추출
3. **Subagent Delegation**: Mickey가 orchestrator로서 목적별 subagent에 작업 위임, 병렬 처리

이 세 가지가 결합되면, Mickey는 "AI를 잘 쓰는 법"의 다음 단계 — **AI가 스스로 학습하고 협업하는 환경을 설계하는 법** — 을 보여줄 수 있습니다.
