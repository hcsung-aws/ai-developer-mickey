# Domain Profile

## Primary Domains
- AI 에이전트 시스템 설계/개선 (Mickey 자체)
- 게임 서버/클라이언트 도구 개발 (패킷 캡처, 리플레이, QA 자동화)
- AWS 인프라 + AIOps (CDK, Bedrock, AgentCore)
- 업무 자동화 도구 (크롤링, 번역, 봇)

## Decision Style

### 핵심 성향: 점진적 확장 + 검증 기반 진행
- **작게 시작**: PoC/최소 기능부터 동작 확인 → 기능 추가 → 구조 정리 → 프로덕션
- **Phase 분해**: 큰 목표를 Phase→Step으로 나누고, 각 단계 E2E 검증 후 다음 진행
- **검증 우선**: 변경 전 기존 동작을 테스트로 캡처(WELC), 변경 후 E2E 확인. 추측으로 넘어가지 않음
- **실수에서 배움**: 빠르게 사용 가능한 결과물을 만들면서 실패를 통해 개선. 완벽을 기다리지 않음

### 계획과 실행의 균형
- 큰 변경: 상세 계획 문서(IMPROVEMENT-PLAN) 선행 → 다음 세션에서 구현. "계획의 구체성이 실행 속도를 결정"
- 작은/명확한 작업: 빠른 실행 우선 (gamejob_crawler: 2세션에 전체 시스템 구현+배포)
- 판단 기준: 영향 범위와 되돌리기 난이도. 넓고 되돌리기 어려우면 계획 선행, 좁고 되돌리기 쉬우면 즉시 실행

### 외부 기술 채택 방식
- 외부 기술을 그대로 복사하지 않음. 자기 맥락에서 재해석하여 선별 채택
- Gap 분석: "이것이 내 문제를 해결하는가?" → "기존 체계와 어떻게 통합되는가?" → "복잡도 대비 효용이 충분한가?"

### 자율성과 피드백
- 자율성의 핵심은 범위가 아니라 피드백 루프의 품질
- AI에게 자율 실행을 허용하되, 결과를 검증하고 교정하는 루프가 반드시 존재해야 함
- 시간이 지날수록 확인 빈도는 줄이되, 정기적 포스트모템으로 품질 유지

### 엔트로피 관리
- 축적된 복잡도를 주기적으로 정리 (아카이빙, 승격, 은퇴)
- 문서/규칙도 팽창→수축 사이클을 거침

## Relationship Preferences

### 연관으로 판단하는 기준 (우선순위 순)
1. **같은 결정 패턴을 공유**: 서로 다른 프로젝트/맥락에서 같은 성향(점진적, 검증 기반 등)이 적용된 사례
2. **위반 시 실패한 사례**: 성향을 따르지 않았을 때 발생한 문제 ↔ 해당 성향 원칙
3. **도구/기법이 같은 목적을 서비스**: WELC, E2E 테스트, Characterization test → 모두 "검증 기반 진행"을 서비스
4. **한 지식이 다른 지식의 전제/확장**: Phase 분해 → 각 Phase 내 WELC 적용 (applies-to)
5. **실전 프로젝트 ↔ 프로토콜 개선**: 프로젝트 실패가 Mickey 프로토콜 개선으로 연결된 경로

### 연관으로 판단하지 않는 것
- 단순히 같은 기술 스택을 사용 (C#이라고 다 연관은 아님)
- 시간적으로 가까움 (같은 날 작업했다고 연관은 아님)
- 표면적 키워드 유사성 (깊은 맥락 없는 단어 매칭)
